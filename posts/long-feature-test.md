# 一篇用来测试博客功能的长文章

这是一篇故意写得比较长、结构比较完整的测试文章。它不是为了表达一个特别宏大的观点，而是为了把博客里常见的阅读场景都跑一遍：长标题、短标题、目录跳转、阅读进度、图片显示、代码块复制、列表、引用、搜索关键词、标签筛选，以及在页面来回切换时的过渡效果。

如果你正在检查这个网站，可以从头慢慢往下滑。右侧目录应该会随着文章标题展开，阅读进度条应该在你滚动时变化；图片应该能正常加载，代码块应该保持清晰，不会把页面撑坏。

![深灰色磨砂玻璃写作桌面](./assets/uploads/test-long-post-desk.svg)

## 为什么需要一篇测试长文

博客在只有两三篇短文时，看起来往往没有问题。首页卡片不会太多，文章页不会太长，目录也只有一两个项目，很多潜在的细节都藏起来了。真正的问题一般会在内容变长之后出现，比如行高是否舒服、图片是否太重、代码块是否溢出、右侧目录是否能跟得上滚动。

一篇测试文章的价值，就是把这些问题提前暴露出来。它像一次温和的压力测试，不追求破坏网站，只是把真实使用里经常出现的东西集中到同一篇文章里。

### 测试时可以重点看什么

- 文章页打开时是否有明显卡顿。
- 右侧目录是否能看到 H1 到 H4 的层级。
- 点击目录里的小标题后，正文是否跳到正确位置。
- 图片加载时页面是否突然抖动。
- 代码块是否能横向滚动或完整显示。
- 页面返回主页、再进入文章页时，过渡是否自然。
- 搜索这篇文章的关键词，比如“磨砂玻璃”“代码块”“图片测试”，是否能命中。

### 一个简单判断

如果你从头到尾读下来，没有明显觉得“哪里挡住了我”，这篇文章就完成了它的任务。博客的界面不需要每一秒都展示存在感，它最重要的是让文章本身变得安静、稳定、可读。

## 页面结构测试

下面这一节会故意放入多级标题。你可以看右侧目录是否按层级缩进，也可以测试点击 H2、H3、H4 时页面滚动是否准确。

### H3：阅读节奏

一篇文章最怕的是所有内容都堆在一起。标题不只是为了好看，它也是读者的呼吸点。尤其是长文，读者经常不是一次读完，而是来回跳着看。标题层级越清晰，越容易让人知道自己在哪里。

#### H4：一个更细的小节

这里是 H4，用来测试最深一级的目录缩进。它不应该抢过 H2 和 H3 的视觉权重，也不应该小到看不清。理想状态是：读者能看见它，但不会被它打断。

### H3：段落密度

段落密度决定了长文是否耐读。如果一段文字太长，眼睛会累；如果每一句都单独成段，文章又会显得很碎。这个博客现在偏向安静的阅读风格，所以段落可以稍微舒展一点，行高也应该留出足够空间。

在深色磨砂玻璃风格里，文字颜色尤其重要。纯白文字会刺眼，太淡又看不清。最舒服的状态通常是标题更亮，正文略微柔和，辅助信息更克制。这样页面会有层次，但不会乱。

## 图片测试

图片是静态博客里最容易拖慢页面的东西。真正上线后，建议尽量使用压缩后的 WebP 或者尺寸合适的图片，不要把手机原图直接塞进文章。对于示意图、流程图、简单装饰图，SVG 往往很适合，因为它体积小，而且缩放后依然清楚。

![代码测试面板](./assets/uploads/test-long-post-code.svg)

上面这张图是轻量 SVG，用来模拟文章中的代码截图或设计示意图。它应该随着文章宽度自适应，不应该撑破卡片，也不应该加载很久。

### 图片在文章中的位置

图片最好出现在真正需要它的地方，而不是都堆在文章末尾。读者看到一段说明，紧接着看到图，就能自然建立联系。如果图和文字离得太远，读者需要来回翻，阅读体验就会下降。

### 图片命名

图片文件名也建议保持清楚，比如 `post-slug-cover.webp`、`post-slug-diagram-01.webp`。这样以后文章多了，维护起来不会乱。现在这篇测试文章使用的是：

- `test-long-post-desk.svg`
- `test-long-post-code.svg`
- `test-long-post-flow.svg`

## 代码块测试

下面放几段不同类型的代码，用来测试代码块显示、复制按钮、换行和横向滚动。

### JavaScript 示例

```js
const posts = [
  { title: "静态博客系统", tags: ["GitHub Pages", "Markdown"] },
  { title: "安静的写作流程", tags: ["Writing", "Focus"] },
  { title: "图片性能优化", tags: ["Performance", "Images"] },
];

function searchPosts(keyword) {
  const query = keyword.trim().toLowerCase();
  if (!query) return posts;

  return posts.filter((post) => {
    const haystack = `${post.title} ${post.tags.join(" ")}`.toLowerCase();
    return haystack.includes(query);
  });
}

console.log(searchPosts("markdown"));
```

这段代码用于测试普通 JavaScript 代码块。重点看缩进、标点和长行是否清楚。

### CSS 示例

```css
.glass-panel {
  border: 1px solid rgba(255, 255, 255, 0.18);
  border-radius: 24px;
  background:
    linear-gradient(145deg, rgba(255, 255, 255, 0.18), rgba(255, 255, 255, 0.06)),
    rgba(255, 255, 255, 0.08);

    0 28px 70px rgba(0, 0, 0, 0.28),

  backdrop-filter: blur(28px) saturate(160%);
  -webkit-backdrop-filter: blur(28px) saturate(160%);
}
```

这段代码可以测试 CSS 代码块在深色界面里的对比度。代码块应该像一个安静的工具区域，而不是像另一个突兀的黑色大盒子。

## 长段落测试

接下来是一段稍微长一点的文字，用来测试连续阅读时的舒适度。一个个人博客最难得的不是功能堆得多，而是你愿意长期打开它。它应该让写文章这件事变得顺手，让读文章这件事变得安静。搜索、标签、归档、图片、代码块、登录管理，这些功能都很重要，但它们最终都应该退到内容背后。好的界面像一张整理好的桌子，你知道笔在哪里、纸在哪里、灯在哪里，但你不会一直盯着桌子本身。

磨砂玻璃风格尤其需要克制。它很容易做得“很炫”，也很容易变得乱。真正耐看的玻璃感不是堆很多透明层，而是控制好边界、阴影、模糊、反光和背景之间的关系。背景太花，玻璃就会脏；边框太亮，玻璃就会假；阴影太重，页面就会显得沉。现在这个博客适合走更安静的方向：深灰、细边、轻微高光、少量手绘白色痕迹作点缀。

### 引用测试

> 一个博客不一定要很热闹。它可以只是一个慢慢整理想法的地方，干净、稳定、能长期维护。

引用块应该有明显区别，但不要像警告框一样抢眼。它最好像在纸边轻轻折了一下，让读者知道这句话可以稍微停一停。

### 图片之后的继续阅读

图片之后的正文也很重要。有些页面在图片后会出现很大的空白，或者图片下方紧贴文字，都会显得粗糙。理想状态是图片像一个自然的段落，前后都有呼吸空间。

## 列表测试

下面是一个较长的列表，用来测试项目符号、行距和长文本换行。

- 首页应该能快速看到最新文章、精选文章和标签。
- 文章页应该能稳定显示目录、阅读进度、上一篇下一篇。
- 写作页应该能保存草稿、发布文章、插入图片、调整图片位置。
- 管理页应该能编辑、删除、修改分类和标签。
- 图片上传后应该尽量压缩，避免 GitHub 仓库越来越大。
- 登录功能应该只保护写作和管理，不要影响普通读者阅读。
- 页面过渡应该轻，不要挡住交互太久。
- 鼠标特效应该好看，但不能影响点击。

### 继续加一点文字

如果列表很长，页面仍然应该保持清晰。每一项之间不需要太大的距离，但也不能挤在一起。读者扫一眼就能分辨每一条，这比复杂的装饰更重要。

## 搜索关键词测试

这一节故意放一些方便搜索的关键词：磨砂玻璃、阅读进度、代码块、图片测试、目录跳转、长文章、GitHub Pages、Markdown、WordPress 写作体验、静态博客、云端发布。

你可以回到主页，在搜索框里输入这些词，看看这篇文章是否出现。搜索结果不一定要做得像大型网站一样复杂，但至少应该让读者能按标题、摘要、标签和正文内容找到文章。

## 性能测试

长文章最容易暴露性能问题。页面如果每次滚动都重新计算大量布局，就会卡；图片如果太大，就会慢；编辑器如果每输入一个字都重排整篇文章，也会抖。静态博客的优势是结构简单，所以我们应该尽量利用这个优势。

### 可以继续优化的方向

以后如果你继续加很多图片，可以考虑这些做法：

- 上传前自动压缩成 WebP。
- 为文章图片生成固定宽高，减少加载时的布局跳动。
- 给图片加懒加载，让屏幕外的图片晚一点加载。
- 管理页显示图片体积，提醒哪些图片太大。
- 定期清理没有被任何文章引用的图片。

## 结尾

这篇测试文章到这里差不多结束了。它包含多级标题、图片、代码块、引用、列表、长段落和搜索关键词，应该足够用来检查博客的主要阅读功能。

如果你能顺畅地打开它、滚动它、点击目录、复制代码、看到图片，并且返回主页后页面没有明显卡顿，那么这个博客的阅读基础就算比较稳了。后面再继续打磨，就可以更多关注写作体验和发布流程。
